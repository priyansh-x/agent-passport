How it works

Three verbs. That's the whole API.

Issue a passport, authorize actions against it, delegate to sub-agents with narrower scope.

I
Human delegates
"Book me a flight under $500" — intent becomes a scoped passport.
II
Passport issued
Ed25519-signed token with permissions, budget, and TTL baked in.
III
Every action checked
Before each tool call, authorize() runs the action against the policy engine.
IV
Deny or allow
Out of scope? Over budget? Expired? Denied. Everything is logged to the audit trail.
Issue
Create a signed passport scoped to specific actions, budgets, and time windows.
const passport = AgentPassport.issue({
  agent: "booking-bot",
  permissions: ["calendar:*"],
  limits: { maxSpend: 500 },
})
Authorize
Check every action before executing. Allowed or denied — every call is logged.
passport.authorize("calendar:write")
// → ✓ allowed

passport.authorize("database:drop")
// → ✗ denied
Delegate
Hand a narrower passport to a sub-agent. Permissions only shrink, never grow.
const child = passport.delegate({
  agent: "email-drafter",
  permissions: ["email:send"],
})

Security model

Guarantees, not guidelines

Every property is enforced cryptographically or at the protocol level.

PropertyHow it's enforced
✓ Tamper-proofEd25519 signatures verified on every authorize() call
✓ No escalationDelegation only narrows permissions — enforced by policy engine + Datalog
✓ Cascade revocationRevoking a parent kills the entire delegation subtree
✓ Replay protection128-bit cryptographic nonce per passport
✓ Budget enforcementServer-side spend tracking — clients can't lie about spending
✓ Time-boundAutomatic expiration — child can never outlive parent
✓ Full audit trailEvery action logged — who, what, when, allowed or denied, and why
Agent tracking

Know exactly what your agents did

Every authorize() call is logged — allowed or denied. Query by passport ID to see the full history of any agent. Persisted in SQLite on the authority server, survives restarts.

Audit trail per agent
→ timestamp — when the action was attempted
→ passport ID — which agent tried it
→ action — what they tried to do (e.g. calendar:write)
→ result — allowed or denied
→ reason — why it was denied (e.g. "exceeds $500 budget")

FAQ

Common questions

What stops an agent from removing the authorize() check from the code? +
Agent Passport doesn't run inside the agent — it runs at the service boundary. The authorize() check lives in your server middleware, your MCP server, or your API gateway — code the agent never touches. The agent sends a request with its passport token in the header. Your server validates it. If the agent strips the passport or skips the header, your server returns 401 (unauthorized). It's the same pattern as OAuth: you don't trust the client to check its own permissions — the server enforces them. The agent can't "go into the code" because the enforcement code runs on infrastructure the agent doesn't control.
How is this different from OAuth or API keys? +
OAuth tells you who someone is. API keys prove identity. Neither tells you what the agent is allowed to do right now, how much it can spend, or when to cut it off. Agent Passport is the authorization layer — it sits on top of identity and adds scoped permissions, budget caps, time bounds, delegation chains, and cascade revocation. Think: OAuth says "this is booking-bot." Passport says "booking-bot can read calendars and spend up to $500 until midnight."
What happens if an agent tries to escalate its own permissions? +
It can't. The passport is signed with Ed25519 — changing any field (permissions, budget, expiry) breaks the cryptographic signature. The verification step catches it instantly. For delegation chains using Biscuit tokens, restrictions are appended cryptographically — you can only add constraints, never remove them. This is enforced at the protocol level, not application level.
Can I use this without self-hosting? +
Yes. The SDK works entirely locally for development — no server needed. AgentPassport.issue() and passport.authorize() run in-process. For production with persistent audit trails, spend tracking across restarts, and multi-service verification, you run the authority server (npx @passport-agent/server) — self-hosted on your infra. Hosted cloud authority is on the roadmap.
What frameworks does it work with? +
MCP (Model Context Protocol), Express, Fastify, Next.js, LangChain, CrewAI, and A2A (Agent-to-Agent protocol). Each has a dedicated package with framework-native bindings — middleware for HTTP frameworks, tool wrappers for agent frameworks, and Agent Card extensions for A2A. Hover over the integration cards above to see code examples.
Is this open source? What's the license? +
Fully open source under the MIT license. The protocol spec, all 8 packages, examples, and docs are on GitHub. We use an open RFC process for protocol changes — anyone can propose modifications.
